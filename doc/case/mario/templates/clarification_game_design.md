# 澄清模板：游戏策划（玩法与关卡）

用途：用于“澄清问题收集”命令的输出模板。目标是列出能让玩法/关卡工作开工与验收所需的最小外部信息，并给出默认假设与风险。

## 输出要求（结构化）

每个问题请包含：
- 问题：一句话能让用户回答/提供文件
- 为什么必须问：阻塞点/风险点
- 关联交付物/任务：例如 `mvp_playable_demo`、`level_1_design`、`controls_physics`
- 默认假设（若用户不答）：可执行的默认值
- 风险：默认假设可能带来的偏差
- 优先级：P0/P1/P2（P0=不答就无法开工或会大幅返工）

## 建议问题清单（示例）

1. 平台与操控
   - 目标平台：PC/Web/移动？键盘/手柄/触屏？
   - 目标帧率/分辨率要求？
2. MVP范围
   - MVP 是否只做 1 个关卡 + 1 个Boss/敌人类型？
   - 是否需要 UI（计分/生命/暂停/重开）到什么程度？
3. 核心玩法与手感
   - 移动/跳跃手感参考：更偏“硬核/精准”还是“休闲/容错”？
   - 跳跃：是否支持二段跳？跳跃高度是否随按键时长变化？
   - 受击/无敌帧/击退：有无？参数范围？
4. 关卡与内容
   - 关卡数量与长度：1关还是多关？每关目标时长？
   - 地形与机关：仅平台跳跃还是包含移动平台/陷阱/机关？
   - 敌人类型：最少需要哪些（巡逻、飞行、投射物等）？
5. 美术与音效风格（不涉及版权）
   - 风格：像素/卡通/简笔？
   - 素材来源：自制/开源素材包/占位符？是否允许临时占位符？
6. 版权与命名
   - 是否明确：只做“类马里奥”的玩法，不使用任天堂素材/名称/角色？
   - 项目命名、角色命名是否需要避开商标？

## 交付格式建议

- 用 `clarification_questions_game_design.md`（或 `.json`）输出。
- 若用户未答的条目，必须给出默认假设并标记风险等级。
